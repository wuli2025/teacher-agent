# 技术选型决策树

> 实事求是：先问**库多大、问什么、要多准**，再选技术。不要拿一把钥匙开天下所有锁。

## 第一问：库多大？（决定要不要检索）

```
库能整个塞进上下文窗口吗？
├─ 能（几百篇，几十万 token 内）
│   → 别折腾检索，整库长上下文漫游。llmwiki 主场。
│     （可选给模型配 grep/read 工具做 [[Agentic-RAG|agentic 自取]]）
├─ 中（几千~上万篇，吞不全）
│   → 轻骑兵：[[BM25与全文检索|BM25]]+双链 粗筛整篇 → 模型精读漫游
└─ 大（几十万~百万篇，海量）
    → 正规军：[[查询改写]] → [[混合检索与RRF|三路混合]] → [[重排rerank|重排]] → 整篇喂 + [[Agentic-RAG|多轮自取]]
```
依据见 [[长上下文vs RAG之争]]、[[RAG增强llmwiki方案]]。

## 第二问：问的是什么类型？（决定检索策略）

| 问题类型 | 首选 | 页 |
|---|---|---|
| 精确名词/代码/报错/ID | BM25 全文（向量易「意思对字不对」） | [[BM25与全文检索]] |
| 语义近似/换种说法 | 向量检索 | [[向量检索]] |
| 多跳/顺藤摸瓜/「谁引用了」 | 双链 / [[GraphRAG|图谱]] | [[GraphRAG]] |
| 「整个语料主题是什么」全局问 | [[GraphRAG]] global / [[RAPTOR]] | [[GraphRAG]] |
| 问法含糊/多意图 | [[查询改写]] / [[HyDE]] | [[查询改写]] |
| 复杂、需多步 | [[Agentic-RAG]] | [[Agentic-RAG]] |

**通用建议**：几乎所有场景，**[[混合检索与RRF|hybrid]] + [[重排rerank|rerank]]** 是最稳的默认组合。

## 第三问：组件怎么选？

- **嵌入模型** → 中文/多语自托管首选 **BGE-M3** 或 **Qwen3-Embedding**；详见 [[嵌入模型选型]]。
- **向量库** → <1000 万用 pgvector/Qdrant/[[向量数据库选型|LanceDB]]；十亿级用 Milvus/Vespa。详见 [[向量数据库选型]]。
- **重排器** → 中文首选 **BGE-reranker / Qwen3-Reranker**；图省事用 Cohere Rerank API。见 [[重排rerank]]。
- **框架** → 检索用 LlamaIndex，agent 编排用 LangGraph，强治理用 Haystack。见 [[RAG框架选型]]。
- **托管省事** → Bedrock KB / Azure AI Search / Vertex。见 [[托管RAG服务]]。

## 第四问：怎么知道选对了？

**一定要评测**。先建 [[黄金测试集构建|金标集]]，跑 [[RAGAS]] 看 recall / faithfulness，按 [[RAG评测方法]] 的顺序迭代（先修摄入→再调检索→再优生成）。**没评测就永远停在「我觉得」**。

## 给 llmwiki 的「头一刀」建议

你现在只有向量一条腿。**立刻补 [[BM25与全文检索|BM25 全文检索]]**，凑成 hybrid 两条腿，字面精准立竿见影；再加 [[重排rerank|rerank]]。这是投入最小、收益最快的两步。
