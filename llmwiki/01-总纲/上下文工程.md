# 上下文工程 (Context Engineering)

> **该下死功夫学的头一门手艺。** RAG、长上下文、漫游，全是它的分支。模型越强，这门功夫越重。

## 是什么

Anthropic 官方定义（2025-09）：上下文工程是「**在 LLM 推理过程中，策划并维护一组最优 token（信息）的策略集合**」——涵盖 system prompt、工具、外部数据、对话历史，即 prompt 之外的一切。

指导原则：找到「**能最大化期望结果概率的、尽可能小的高信号 token 集合**」。因为 LLM 有 **attention budget（注意力预算）**，塞太多反而退化（见 [[长上下文vs RAG之争]] 的 context rot）。

**Karpathy 的心智模型**：LLM 是 CPU，上下文窗口是 RAM，我们写的是决定「往内存里加载什么」的操作系统。他公开支持用 "context engineering" 取代 "prompt engineering"，因为真正的工作是「**填充上下文窗口这门精细的艺术与科学**」。

## 为什么它取代了 prompt engineering

- prompt engineering 管「怎么写好一句 prompt」；
- context engineering 管「**整个上下文状态**」，而且是**迭代的**——每一轮都要重新决定传给模型什么。

## Anthropic 的四大技术

1. **即时检索 (Just-in-Time Retrieval)** —— 运行时用工具动态加载，而非预先把一切塞进上下文。这正是 [[Agentic-RAG]] 的内核。
2. **压缩 (Compaction)** —— 接近上限时摘要历史，保关键决策、丢冗余。
3. **结构化笔记 (Structured Note-Taking)** —— 在窗口外维护持久记忆（如 `NOTES.md`）。与 [[LLM-Wiki维护方法论]]、[[Agent记忆系统]] 同源。
4. **子智能体 (Sub-Agents)** —— 专门 agent 干聚焦任务，只回浓缩摘要。见 [[深度研究Agent]]。
   - system prompt 用「恰当的抽象高度 (right altitude)」：够具体能引导，够灵活留空间。

## LangChain 四桶框架

把上下文操作归为四类：
- **Write（写出）** —— 存到窗口外：scratchpad、memory。
- **Select（选入）** —— 拉进窗口：检索记忆、**对工具描述做 RAG 只取相关工具（报告称准确率提升 3 倍）**、代码 RAG。
- **Compress（压缩）** —— 递归/层级摘要、剪枝。
- **Isolate（隔离）** —— 多智能体、沙箱、state schema。

## Manus 的实战要点（生产 agent）

- **围绕 KV-Cache 设计**：缓存命中率是生产 agent 最重要的指标（缓存输入比未缓存便宜约 10×）。保持 prompt 前缀稳定、上下文 append-only。
- **屏蔽而非移除工具**：中途增删工具定义会让 KV-cache 全失效。
- **文件系统当无限外部记忆**。
- **Recitation（复述）**：让 agent 不断重写任务摘要把目标推到最近上下文，对抗 lost-in-the-middle。
- **保留错误轨迹**：让模型从失败里隐式学习、不再犯。

## 与 llmwiki 的关系

llmwiki 的「结构化 wiki + 双链」本质就是**为上下文工程预先把知识组织好**：高信号、结构化、可按需加载。同一思想的不同投影还有 [[STORM与Co-STORM|llms.txt]]、[[LLM-Wiki维护方法论|NOTES.md]]、Manus 的文件系统记忆。库主把功夫下在结构上，等于在练这门最值钱的手艺（见 [[RAG与llmwiki核心论点]]）。
