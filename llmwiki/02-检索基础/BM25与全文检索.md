# BM25 与全文检索（倒排索引 / 稀疏检索）

> **该立刻给 llmwiki 补的头一刀。** 你现在只有向量一条腿，补上 BM25 凑成 [[混合检索与RRF|hybrid]] 两条腿，字面精准立竿见影。见 [[RAG与llmwiki核心论点]]「老把式不能丢」。

## 是什么

基于**倒排索引**（词 → 含该词的文档列表）做关键词精确匹配。

- **TF-IDF**：词频 × 逆文档频率，奖励「在本文档常见、在全库罕见」的词。
- **BM25 (Okapi BM25)**：概率检索标准。在 TF-IDF 上加两件事——**词频饱和**（参数 `k1`，重复词收益递减）和**文档长度归一化**（参数 `b`）。**到 2026 仍是最强的字面检索基线。**

## 为什么到今天还香

- 专有名词、代码、人名、报错码、ID —— 向量最容易「意思对字不对」，全文检索一抓一个准。
- **便宜、可解释、可调**，毫秒级、无需 GPU。
- 与向量天生互补（向量懂语义但不懂字面），这正是 [[混合检索与RRF|hybrid]] 存在的理由。

## 工具

- **SQLite FTS5**：内置倒排，`bm25()` 排序，适合本地/嵌入式 KB。缺点：tokenizer 选项少、重写并发弱。
- **Tantivy**（Rust，Lucene 风格）：BM25、faceting、增量索引，比 Lucene 快约 2×，支撑 ParadeDB/Quickwit。
- **Elasticsearch / OpenSearch**（Lucene）：功能全、分布式，原生支持向量 + RRF 混合。

## ⚠️ 中文大坑：分词

中文没有空格词边界，字面检索**必须**配 CJK 分词，否则形同虚设：
- FTS5 默认 unicode tokenizer 对中文很差 → 用 **trigram** tokenizer 或 **jieba** 自定义分词。
- Elasticsearch 用 **IK / smartcn** 分析器。
搭配 **同义词表/查询扩展**（「番茄」↔「西红柿」）几乎零成本补字面死角。

## 与 llmwiki

llmwiki 的双链、反链、规范命名本身也是「结构即检索」的老把式（见 [[LLM-Wiki维护方法论]]）。BM25 是其中最该先补的「术」。召回单位仍是**整篇/整节**。
