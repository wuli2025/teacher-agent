# CLAUDE.md —— 本知识库的维护规范（schema）

本文件告诉任何 AI 助手：**如何把这个库当成一座 [[LLM-Wiki维护方法论|LLM Wiki]] 来维护**，而不是当成一次性聊天。读完照做。

## 这个库是什么

一座关于 **RAG 技术 + 如何与 llmwiki 结合 + 如何自主学习/知识补全** 的方法论知识库。服务对象是库主（一位深耕 llmwiki、走 Karpathy 结构化路线的同志），用途是**系统学习**与**技术选型**。主线见 [[RAG与llmwiki核心论点]]。

## 三层结构（Karpathy 模式）

1. **raw（原始来源）** —— 此库暂不存原文，外部来源统一登记在 [[来源汇总]]。
2. **wiki（本库主体）** —— `00`~`07` 目录下的 markdown 页，由 AI 生成与维护。
3. **schema（本文件）** —— 约定、命名、工作流。

## 页面写法约定

- **每页一个主题（原子化）**。文件名 = 主题名（中文，必要时中英并列），与 `[[双链]]` 内的名字一致。
- **frontmatter** 用于检索与分类：
  ```yaml
  ---
  title: 标题
  aliases: [别名, English Name]   # 方便双链与检索命中
  tags: [retrieval, advanced, ...]
  type: concept | architecture | engineering | eval | integration | nav | memory
  status: stable | draft
  updated: YYYY-MM-DD
  ---
  ```
- **正文结构建议**：`是什么 → 怎么干 → 何时用 / 优劣 → 与 llmwiki 怎么配 → 关联 → 来源`。
- **双链 `[[…]]` 要连得密**。结构本身就是检索（见 [[RAG与llmwiki核心论点]]）。链到尚不存在的页也行——那是「待补的坑」，登记进 [[知识补全与缺口检测]]。
- **来源要标**。一手优先（arXiv / 官方博客 / 官方文档）。二手数字（如「准确率 86% vs 32%」「6000× token」）注明「待核」。
- **区分事实与推断**。技术判断、选型建议属推断，明确标出，不伪托。

## 三个动作（ingest / query / lint）

- **ingest（吸收新知识）**：读新材料 → 写/更新对应主题页 → 更新 [[index]] 一行摘要 → 在 [[log]] 追加一条 → 顺手补相关页的双链。
- **query（回答提问）**：先搜本库已编译好的页，再合成答案并带 `[[页名]]` 出处；有价值的新分析**回填**成新页或补进既有页。
- **lint（健康检查）**：定期扫描——孤儿页、断链（`[[X]]` 但无 X 页）、被多次提及却无专页的概念、过时声明、缺来源。结果记入 [[知识补全与缺口检测]]。

## log 约定

[[log]] 为 append-only，统一前缀便于解析：
`## [YYYY-MM-DD] ingest|lint|fix | 一句话`

## 风格

库主偏好「实事求是、一分为二」的判断口吻：给结论，也给适用边界；不把任何技术供成「放之四海皆准的死公式」。
