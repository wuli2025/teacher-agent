# 向量检索（语义检索 / Dense Retrieval）

> llmwiki 已有这条腿（LanceDB，召回 58%→71%）。它管「意思相近」，缺点是字面不精准——所以要补 [[BM25与全文检索|BM25]] 凑 [[混合检索与RRF|hybrid]]。

## 是什么

用嵌入模型把 query 和文档编码成稠密向量（256~4096 维），语义相近的文本向量也相近。检索 = 给 query 编码后，找最近的文档向量。

## 怎么干

- **嵌入**：bi-encoder（双编码器），query 和文档独立编码，文档向量可离线预算好建索引。见 [[嵌入模型选型]]。
- **相似度**：余弦相似度（看夹角，不看长度）或点积（向量归一化后等价，更快）。
- **ANN 近似最近邻**（暴力 O(N) 太慢）：
  - **HNSW**（分层可导航小世界图）：快、召回高，是多数向量库默认。代价：内存大、动态增删贵。
  - **IVF**（倒排文件）：k-means 聚类，只搜最近的簇（`nprobe` 调召回/速度）。省内存，适合大静态集。
  - **PQ**（乘积量化）：压缩向量省内存提速，损一点精度，常和 IVF/HNSW 组合。

## 优劣

- **强**：同义、改写、概念匹配；多语模型可跨语言。
- **弱**：精确 token（报错码、ID、专有名词、SKU）——容易「意思对、字不对」；存算随维度增长。

## 与 llmwiki

向量是 [[RAG与llmwiki核心论点|侦察兵]] 的一条腿，**不是主帅**。用它做语义粗召回，但召回单位要是**整篇/整节**（见 [[分块策略]]），别交一堆碎片。中文场景务必和 [[BM25与全文检索|BM25]] 合用，因为技术名词、代码靠向量抓不准。
