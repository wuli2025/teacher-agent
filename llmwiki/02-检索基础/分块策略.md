# 分块策略 (Chunking)

> 核心立场（与 [[RAG与llmwiki核心论点|主线]] 一致）：**召回的单位是「整篇/整节」，不是撕碎的 chunk。** 碎片保不住前后文，整篇才保得住结构——这是和老 RAG 最大的分界线。

## 基础切法

- **定长 (fixed-size)**：每 N token + X% 重叠。简单快，但盲切。常用 200–500 token、10–20% 重叠。
- **递归 (recursive)**：按分隔符优先级（段→句→词）切，兼顾结构与大小。**通用强默认**。
- **句子 (sentence)**：便宜，意外地能打。
- **语义 (semantic)**：按相邻句相似度骤降处切（话题切换）。直觉好。

## ⚠️ 2026 的「降温」证据

- NAACL 2025：**定长 ~200 词常追平甚至胜过语义切块**，额外算力常不值。
- 但临床等结构化领域，话题对齐切块远超定长（87% vs 13%）。
- **结论**：**没有单一最优切法**。把劲花在检索+重排+情境注入上，别死磕花式切块（呼应 [[学习路线图]]「别死磕」档）。

## 「搜小、返大」三件套（重点）

把**搜索单元**和**喂给模型的单元**解耦——搜小（精准），返大（语境足）：
- **父文档 / 小到大 (parent-document / small-to-big)**：索引小子块做精准匹配，命中后返回**父块**（段/节/页/整篇）。LlamaIndex 的 auto-merging 是层次变体。
- **句子窗口 (sentence-window)**：索引单句，命中后返回该句 + 前后 N 句。
- **命题切块 (proposition)**：拆成原子自足陈述（解消代词）。精准但预处理贵，常配父文档。

## 两把「情境注入」利器（提升块向量质量）

- **[[上下文检索-Contextual-Retrieval]]**（Anthropic）：给每块前置一段 LLM 生成的情境，再嵌入 + 建 BM25。改 dense+sparse 双索引，~$1/M token。
- **[[Late-Chunking]]**（Jina）：先整篇编码出 token 向量，再切块时对块内 token 池化。只改 embedding，但更便宜。

## 为什么「整篇 > 碎片」

- 碎片丢语境与指代（「这修了那个 bug」——哪个 bug？）；
- 答案常跨边界（完整流程、整段配置、代码+解释）；
- 定长会在恰好相关处切断；
- 现在长上下文又大又便宜，返整节代价小、收益大。

## 与 llmwiki

llmwiki 的文章本身就是「整篇」单元，天然契合「搜小返大」。给模型的应是**整篇好结构**，让它自己 [[Agentic-RAG|漫游]]。这也是为什么 [[RAG与llmwiki核心论点|功夫下在结构上]] 比死磕切块更值。
