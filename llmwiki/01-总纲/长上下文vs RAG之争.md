# 长上下文 vs RAG 之争

> 大模型窗口越来越大，是不是「RAG 已死」？**对一半。** 见 [[RAG与llmwiki核心论点]]。

## 争论焦点

窗口从几千字撑到几十万、上百万 token。一派喊「直接把整库塞进去，RAG 没用了」；另一派说「再大也吞不下海量库，且长上下文又贵又慢又会退化」。

## 长上下文会退化的硬证据

- **Lost in the Middle**（Liu et al., TACL 2024）：相关信息在上下文**开头或结尾**时表现最好，**中间显著退化**；上下文越长整体越差。
- **RULER**（NVIDIA, 2024）：提出 **有效上下文长度 (effective context length)**——模型真正能维持高性能的长度，往往**远低于宣称窗口**。很多号称 32K+ 的模型在 32K 已退化。
- **Context Rot**（Chroma, 2025-07，实测 18 个前沿模型）：「**性能随输入长度显著变化，即便简单任务也是**」。1M 窗口模型常在 ~300K–400K token 出现明显退化；干扰项 (distractors) 会非均匀加剧退化。
- 早期研究：**4K 上下文 + 检索可胜过 16K 上下文无检索**。

## 成本与延迟

- RAG 平均查询 ~1 秒；纯长上下文可达 30–60 秒。
- LLM 按输入 token 计费，长上下文成本随 token **线性增长**。
- 缓解手段见 [[成本与延迟优化]]（[[成本与延迟优化|prompt 缓存]]、语义缓存）。

## 2026 主流共识：互补，不是替代

- 「RAG is dead」是被夸大的口号。百万窗口没让 RAG 过时。
- **赢家是混合 (Hybrid)**：检索先**缩小搜索空间、控成本延迟**，长上下文再在**精选证据上做深度综合**。学术代表 **SELF-ROUTE**（Xu et al., 2024）——让模型自判该走 RAG 还是长上下文。
- 选哪个取决于**规模、延迟预算、更新频率、成本**，不是教条二选一。

## 对 llmwiki 的落点

这正是 [[RAG增强llmwiki方案]] 的分阶段配方：
- **小库**（几百篇，吞得下）→ 整库长上下文漫游，别折腾检索。llmwiki 的主场。
- **中库** → BM25 + 双链粗筛整篇 → 模型精读漫游。
- **大库** → [[查询改写]] → [[混合检索与RRF|三路混合]] → [[重排rerank|重排]] → 整篇喂 + [[Agentic-RAG|多轮自取]]。

**关键**：不管哪阶段，最后落点都是「把整篇好结构交给模型自己读」，而不是喂碎渣。窗口变大只是让「装不下」那条线往后退，**没改变「先筛后喂」这个魂**。
