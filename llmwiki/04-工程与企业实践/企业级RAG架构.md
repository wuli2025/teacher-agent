# 企业级 / 生产级 RAG 架构

> 「摊子是汪洋大海」时的正规军打法。2026 共识：**hybrid 检索 → 重排 → 自适应生成**，且 ~80% 失败源于摄入/切块而非 LLM。

## 参考架构（模块化 Modular RAG）

1. **摄入 (Ingestion)**：解析 → [[分块策略|切块]] → 元数据抽取 → 嵌入。（失败重灾区）
2. **索引 (Indexing)**：向量 + 字面([[BM25与全文检索|BM25]]) + 丰富元数据；常并存 [[GraphRAG|知识图谱]]。
3. **检索 (Retrieval)**：[[混合检索与RRF|hybrid]] top-50 → 元数据/安全过滤 → [[重排rerank|重排]] top-5。
4. **生成 (Generation)**：grounded 上下文 + [[Adaptive-RAG与路由|按问题复杂度路由]]。
5. **观测 (Observability)**：全链路可追踪、可量化、可调试 + 持续 [[RAG评测方法|评测]]。

## 规模化到百万/十亿文档

- **索引算法**：HNSW 主导；十亿级分片难（图连通→跨片 RPC）。
- **磁盘+压缩**：DiskANN/ScaNN；**乘积量化省内存 ~90%**（损召回）。
- **分布式**：分片+复制；十亿级用 Milvus 分布式或 Vespa（见 [[向量数据库选型]]）。

## 多租户 / 访问控制 / 时效

- **隔离模型**：**Silo**（每租户独立索引，隔离最强，企业默认）/ **Pool**（共享索引 + 租户 ID 过滤，省成本）/ **Bridge**（混合）。
- **访问控制**：**检索前过滤 (pre-retrieval filtering)**——在送进 LLM 前就按租户/用户元数据过滤；运行时过滤键须匹配索引元数据键。
- **治理**：索引前分级/敏感标注、血缘追踪、**新鲜度评分**标记陈旧源；增量/delta 重索引保时效。

## 与 llmwiki

llmwiki 通常是**个人/团队尺度**，多数企业级重武器用不上。但有三点可借：
- **摄入质量决定一切**——把功夫下在写好结构（[[RAG与llmwiki核心论点]]），等于从源头降低 80% 失败；
- **增量索引/新鲜度**——llmwiki 用 git 天然有版本与增量；
- 若将来多人/多团队用，可参考 Silo/Pool 隔离与检索前过滤做权限。
