# 成本与延迟优化

> 2026 延迟标准：**亚秒级**（2024 还是 ~3 秒）。这页是把 RAG 跑得又快又省的工具箱。

## 主要手段

- **语义缓存 (semantic cache)**：对语义相近的重复查询直接命中，可吸收约 **40%** 重复查询，命中即免 LLM 调用——省成本省延迟。
- **prompt 缓存**：缓存大段静态上下文（如整篇文档），逐块/逐轮复用便宜。Anthropic [[上下文检索-Contextual-Retrieval]] 预处理靠它降到 ~$1.02/百万 token。也是 [[上下文工程|Manus KV-cache]] 思想的核心。
- **prompt 压缩**：LLMLingua 式压缩长检索上下文，质量损失小；先「收紧 prompt + 约束输出」。
- **嵌入量化 (quantization)**：scalar = 16× 压缩、binary = 32× 压缩，1024+ 维模型仍保 ~92–96% 性能。配 ANN + 降维（[[嵌入模型选型|Matryoshka]]）做实时。
- **召回-重排两段**：[[混合检索与RRF|hybrid]] 融合仅 ~6ms，远低于 LLM 500ms–2s；[[重排rerank|重排]] 只排 top 50–150。

## 长上下文的成本现实

纯长上下文每查可达 30–60 秒、按 token 线性计费（见 [[长上下文vs RAG之争]]）。所以「整库塞进去」在大库**不可持续**——这正是要 [[RAG与llmwiki核心论点|侦察兵先筛]] 的经济理由。

## 与 llmwiki

- 小库漫游时，用 **prompt 缓存**把整库/常用页缓存，重复提问几乎零边际成本；
- 中大库，靠「[[混合检索与RRF|hybrid]] 粗筛整篇 → [[重排rerank|重排]] → 只喂 top」控住喂进窗口的 token；
- 增量更新（git）避免全量重嵌入。
