# 查询改写 (Query Transformation)

> 「先把问题问清楚，再去找。」用户的话往往含糊，检索前优化问题，比拿原话去探准得多。[[RAG增强llmwiki方案|侦察兵五条]]之第四。

## 是什么

检索前用 LLM 改写/扩展/拆解用户问题。检索质量上限取决于查询质量。

## 主要手法

- **查询改写 (Query Rewrite)**：把口语化/多轮对话里的问题理顺（消解代词、补上下文）。多轮聊天必备。
- **查询扩展 (Query Expansion)**：加同义词/相关词扩大召回（经典 IR 技术，也可用 LLM）。
- **多查询 (Multi-Query)**：LLM 生成几个改写版本，各自检索，并集去重（常接 [[混合检索与RRF|RRF]]）。提召回、抗措辞敏感。
- **退一步提问 (Step-back)**：先生成更抽象的版本，检索背景/原理，再答。适合字面问题太窄、匹配不到解释段落。
- **子问题分解 (Decomposition)**：把复杂/多跳问题拆成独立子问，各自检索再综合。适合比较类、多部分问题。
- **[[HyDE]]**：让 LLM 先写假设性答案，用答案去检索（答案比问题更像目标文档）。

## 优劣

都增加 LLM 延迟/成本，且有「改歪了反而更糟」的风险。可组合（分解 → 每个子问 multi-query → 每路 HyDE）。**按问题类型选择性使用**。

## 与 llmwiki

[[Agentic-RAG|agentic]] 模式里，查询改写天然内化为模型「自己想清楚再 grep/跳链」的能力——不必固定成后台一步。小库直接漫游时常不需要；大库正规军阶段值得上。
