# RAG 失败模式（七大失败点 + 修法）

> 论文《Seven Failure Points When Engineering a RAG System》(Barnett et al., arXiv 2401.05856)。结论：**RAG 的鲁棒性是「运行中演化」出来的，不是一次设计成的。**

## 七大失败点

| # | 失败点 | 现象 | 修法 |
|---|---|---|---|
| FP1 | **缺内容** | 答案不在库里，却硬编 | 让模型会说「不知道」（[[评测基准|negative rejection]]）；[[知识补全与缺口检测\|补库]]；[[CRAG\|联网兜底]] |
| FP2 | **漏掉头部文档** | 对的文档存在但没进 top-k | 提召回：[[混合检索与RRF\|hybrid]]、[[查询改写]]、调 k |
| FP3 | **整合时丢失** | 召回了但没进上下文窗口 | 控候选数、[[重排rerank\|重排]] 后精选 |
| FP4 | **没抽取出** | 在上下文里但噪声/冲突盖过 | [[重排rerank\|重排]]、收紧 top-k、降噪 |
| FP5 | **格式错** | 没按要求的表格/列表 | prompt 约束输出格式 |
| FP6 | **具体度不当** | 太泛或太细 | prompt 调粒度、[[Adaptive-RAG与路由\|路由]] |
| FP7 | **不完整** | 信息够却只答一半 | 子问题分解（[[查询改写]]）、[[Agentic-RAG\|多轮]] |

补充实战坑：**context pollution（上下文污染）**——太多无关文档稀释信号 → [[重排rerank|重排]] + 收紧 top-k。

## 关键启示

- **验证只能在运行中做**；上线后持续收集失败案例回灌 [[黄金测试集构建|金标集]]；
- ~80% 失败在**摄入/[[分块策略|切块]]**层——先修这里（见 [[RAG评测方法]] 迭代顺序）。

## 与 llmwiki

llmwiki 的几个失败对应物与修法：
- FP1 缺内容 → 触发 [[自主学习工作流|自主补库]]；
- FP2/FP3 → 召回**整篇**（[[分块策略]]）+ 双链漫游天然减少「漏关键页」；
- 整体 → 把结构写好（[[RAG与llmwiki核心论点]]）从源头减少多数失败点。
